## <font style="color:#FF0000;">【强制】</font>避免通过 Intent 传递大数据
### 说明
+ Android 中组件间的数据传递通常使用 `Intent` + `Parcelable` 或 `Serializable`。
+ Binder 机制对每次 Transaction 的数据大小有限制，约 **1MB**。
+ 如果传递的数据过大，可能会触发 `TransactionTooLargeException` 或导致 OOM，应用崩溃。

### 正例（推荐做法）
1. 使用 **轻量数据**：

```java
Intent intent = new Intent(this, NextActivity.class);
intent.putExtra("userId", 12345);  // 仅传递必要信息
startActivity(intent);
```

2. 使用 **事件总线（EventBus、LiveData 等）** 传递大对象：

```java
// 通过 EventBus 发布大对象
EventBus.getDefault().postSticky(largeDataObject);
```

3. 使用 **数据持久化**：
+ `SharedPreferences`、`SQLite`、`Room` 等存储大数据；
+ 目标组件再去读取。

### 反例（错误做法）
```java
Intent intent = new Intent(this, NextActivity.class);
intent.putExtra("largeData", largeBitmap); // Bitmap 或大对象
startActivity(intent); // 可能抛出 TransactionTooLargeException
```

✅ 要点：

1. 避免 Intent 传递超过 1MB 的数据；
2. 大对象可通过持久化或全局事件机制传递；
3. 对于 Parcelable 对象，尽量只传递必要字段。

## <font style="color:#FF0000;">【强制】</font><font style="color:#000000;">Application 初始化需做进程判断</font>
### 说明
+ Android 应用可能包含 **多个进程**（主进程、服务进程等）。
+ Application#onCreate 会在每个进程中调用一次，如果在所有进程都执行业务初始化，会浪费资源并可能导致异常。
+ 建议在 Application 初始化时根据进程判断，只在目标进程执行对应业务逻辑。

### 正例
```java
public class MyApplication extends Application {
    @Override
    public void onCreate() {
        super.onCreate();

        String currentProcess = getCurrentProcessName();

        // 所有进程通用初始化
        initCommon();

        // 仅在主进程初始化
        if (isMainProcess(currentProcess)) {
            initMainProcess();
        }

        // 仅在后台服务进程初始化
        if (isBgProcess(currentProcess)) {
            initBgProcess();
        }
    }

    private String getCurrentProcessName() {
        int pid = android.os.Process.myPid();
        ActivityManager am = (ActivityManager) getSystemService(Context.ACTIVITY_SERVICE);
        for (ActivityManager.RunningAppProcessInfo processInfo : am.getRunningAppProcesses()) {
            if (processInfo.pid == pid) {
                return processInfo.processName;
            }
        }
        return null;
    }

    private boolean isMainProcess(String processName) {
        return getPackageName().equals(processName);
    }

    private boolean isBgProcess(String processName) {
        return "com.peakmain.app:service".equals(processName);
    }
}
```

### 反例
```java
public class MyApplication extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        // 在所有进程都执行业务初始化，浪费资源
        initMainProcess();
        initBgProcess();
    }
}
```

要点：

1. Application#onCreate 在每个进程都会执行；
2. 业务初始化必须根据进程判断，避免无效初始化；
3. 对于后台进程尽量减少耗时操作和无关逻辑。

## <font style="color:#FF0000;">【强制】</font><font style="color:#000000;">新建线程必须通过线程池</font>
### 说明
+ **禁止直接使用 new Thread() 显式创建线程**。
+ 推荐使用线程池（AsyncTask、ThreadPoolExecutor 或自定义线程池）。
+ 线程池的好处：
    1. **复用线程**：减少线程创建和销毁的开销，提高性能；
    2. **控制资源**：避免因频繁创建线程导致系统资源耗尽或内存不足；
    3. **方便管理**：便于任务调度、异常处理和性能分析。

### 正例
```java
int NUMBER_OF_CORES = Runtime.getRuntime().availableProcessors();
int KEEP_ALIVE_TIME = 1;
TimeUnit KEEP_ALIVE_TIME_UNIT = TimeUnit.SECONDS;
BlockingQueue<Runnable> taskQueue = new LinkedBlockingQueue<>();

ExecutorService executorService = new ThreadPoolExecutor(
    NUMBER_OF_CORES,
    NUMBER_OF_CORES * 2,
    KEEP_ALIVE_TIME,
    KEEP_ALIVE_TIME_UNIT,
    taskQueue,
    new BackgroundThreadFactory(),
    new ThreadPoolExecutor.AbortPolicy() // 或自定义拒绝策略
);

// 执行任务
executorService.execute(new Runnable() {
    @Override
    public void run() {
        // 任务逻辑
    }
});
```

### 反例
```java
new Thread(new Runnable() {
    @Override
    public void run() {
        // 任务逻辑
    }
}).start(); // ❌ 不允许直接创建线程
```

要点：

+ 任何线程创建必须通过线程池完成；
+ 禁止匿名线程、手动 new Thread 的方式；
+ 对异步任务的调度和管理应统一通过线程池进行。

## <font style="color:#FF0000;">【强制】</font><font style="color:#000000;">线程池禁止使用 Executors 创建</font>
### 说明
+ **禁止**使用 Executors 系列方法（如 newFixedThreadPool、newSingleThreadExecutor、newCachedThreadPool、newScheduledThreadPool）创建线程池。
+ 原因：
    1. FixedThreadPool 和 SingleThreadPool：内部请求队列长度为 **Integer.MAX_VALUE**，可能堆积大量任务，导致 **OOM**。
    2. CachedThreadPool 和 ScheduledThreadPool：允许创建线程数量为 **Integer.MAX_VALUE**，可能一次性创建大量线程，导致 **OOM**。
+ 建议使用 **ThreadPoolExecutor** 明确线程池的核心参数和拒绝策略，从而控制资源，规避潜在风险。

### 正例
```java
int NUMBER_OF_CORES = Runtime.getRuntime().availableProcessors();
int KEEP_ALIVE_TIME = 1;
TimeUnit KEEP_ALIVE_TIME_UNIT = TimeUnit.SECONDS;
BlockingQueue<Runnable> taskQueue = new LinkedBlockingQueue<>();

ExecutorService executorService = new ThreadPoolExecutor(
    NUMBER_OF_CORES,                 // core pool size
    NUMBER_OF_CORES * 2,             // maximum pool size
    KEEP_ALIVE_TIME,
    KEEP_ALIVE_TIME_UNIT,
    taskQueue,
    new BackgroundThreadFactory(),    // 自定义线程工厂
    new ThreadPoolExecutor.AbortPolicy() // 自定义拒绝策略
);

// 执行任务
executorService.execute(new Runnable() {
    @Override
    public void run() {
        // 任务逻辑
    }
});

```

### 反例
```java
// ❌ 不推荐
ExecutorService cachedThreadPool = Executors.newCachedThreadPool();
ExecutorService fixedThreadPool = Executors.newFixedThreadPool(10);
```

要点：

+ 明确线程池参数：核心线程数、最大线程数、任务队列、线程工厂、拒绝策略；
+ 禁止直接使用 `Executors` 工厂方法创建线程池；
+ 避免因线程或任务无限增长导致系统资源耗尽。

## <font style="color:#528135;">【推荐】</font><font style="color:#000000;">谨慎使用多进程</font>
### 说明
虽然 Android 多进程可以降低主进程的内存压力，但多进程带来以下问题，需要谨慎评估：

1. **首次进入新进程页面延时**
    - 当启动一个新进程中的 Activity 时，可能会出现延迟显示的现象，例如黑屏或白屏几秒。
    - 延迟时间和新 Activity 的主题设置有关（如透明主题、全屏主题等）。
2. **Application 实例多次创建**
    - 多进程下，每个进程都会创建独立的 Application 实例。
    - 需要判断哪些模块必须在所有进程中初始化，哪些只需要在主进程初始化，以避免不必要的资源占用和逻辑错误。

### 建议
+ 尽量减少不必要的多进程使用；
+ 对必须多进程的业务，严格控制每个进程的初始化逻辑；
+ 结合进程判断（如 `Process.myPid()` 或 `ActivityManager`）来控制初始化行为。

### 示例
```java
public class MyApplication extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        // 获取当前进程名
        String processName = getProcessName();
        if ("com.example.app".equals(processName)) {
            // 主进程初始化
        } else if ("com.example.app:remote".equals(processName)) {
            // 其他进程初始化
        }
    }

    private String getProcessName() {
        int pid = android.os.Process.myPid();
        ActivityManager am = (ActivityManager) getSystemService(Context.ACTIVITY_SERVICE);
        for (ActivityManager.RunningAppProcessInfo info : am.getRunningAppProcesses()) {
            if (info.pid == pid) return info.processName;
        }
        return null;
    }
}
```

要点：

+ 多进程增加复杂度和延迟，谨慎使用；
+ 确保每个进程初始化逻辑独立、合理；
+ 结合进程判断避免资源浪费和逻辑错误。
