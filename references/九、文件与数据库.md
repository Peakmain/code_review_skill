## <font style="color:#FF0000;">【强制】</font><font style="color:#000000;">不要硬编码文件路径</font>
### 说明
+ Android 提供了内部存储（应用私有目录）和外部存储（用户可访问目录），用于存放应用数据和用户数据。
+ 硬编码路径不仅存在安全隐患，还容易导致适配问题（不同设备路径不同）。
+ 应通过系统 API 获取文件目录进行操作，常用方法如下：

| API 方法 | 用途 |
| --- | --- |
| Context#getFilesDir() | 获取应用内部私有文件目录 |
| Context#getCacheDir() | 获取应用内部缓存目录 |
| Environment#getExternalStorageDirectory() | 获取外部存储根目录（已不推荐） |
| Environment#getExternalStoragePublicDirectory(type) | 获取公共外部存储目录（如 Environment.DIRECTORY_PICTURES） |


### 正例
```java
public File getDir(String albumName) {
    File file = new File(Environment.getExternalStoragePublicDirectory(
            Environment.DIRECTORY_PICTURES), albumName);
    if (!file.mkdirs()) {
        Log.e(LOG_TAG, "Directory not created");
    }
    return file;
}
```

### 反例
```java
public File getDir(String albumName) {
    // 不要硬编码路径
    File file = new File("/mnt/sdcard/Download/Album", albumName);
    if (!file.mkdirs()) {
        Log.e(LOG_TAG, "Directory not created");
    }
    return file;
}

```

要点：

+ 永远不要硬编码文件路径；
+ 使用系统 API 获取目录，保证安全性与适配性；
+ 对于外部存储，注意 Android 10+ Scoped Storage 的权限限制。

## <font style="color:#FF0000;">【强制】</font><font style="color:#000000;">使用外部存储时必须检查可用性</font>
### 说明
+ 外部存储可能不存在或不可读写，例如 SD 卡未挂载或者权限不足。
+ 在对外部存储进行读写操作前，必须检查其状态，以避免 `FileNotFoundException` 或其他异常。
+ 可以通过 `Environment.getExternalStorageState()` 获取存储状态，并根据返回值判断可读写性。

### 正例
```java
// 检查外部存储是否可写
public boolean isExternalStorageWritable() {
    String state = Environment.getExternalStorageState();
    return Environment.MEDIA_MOUNTED.equals(state);
}

// 检查外部存储是否可读
public boolean isExternalStorageReadable() {
    String state = Environment.getExternalStorageState();
    return Environment.MEDIA_MOUNTED.equals(state) || Environment.MEDIA_MOUNTED_READ_ONLY.equals(state);
}
```

要点：

+ 写操作前调用 `isExternalStorageWritable()`
+ 读操作前调用 `isExternalStorageReadable()`
+ 避免在外部存储不可用时直接访问，防止崩溃或异常数据



## <font style="color:#528135;">【推荐】</font><font style="color:#000000;">SharedPreferences 仅用于存储简单数据类型 </font>
### 说明
+ SharedPreferences 适合存储 **简单数据类型**：
    - `int`、`long`、`float`、`boolean`、`String`
+ 不适合存储复杂对象或大量数据，将复杂数据转成 `String` 存储会带来：
    - 数据解析复杂
    - 容易出错
    - 性能消耗大
+ 对于复杂对象或大数据，推荐使用：
    - 文件（JSON、序列化等）
    - SQLite / Room 数据库

### 正例
```java
public void updateSettings() {
    SharedPreferences mySharedPreferences = getSharedPreferences("settings", Activity.MODE_PRIVATE);
    SharedPreferences.Editor editor = mySharedPreferences.edit();
    editor.putString("id", "foo");
    editor.putString("nick", "bar");
    // 不要将复杂数据类型转成String存储
    editor.apply();
}
```

要点：

+ 仅用于简单配置或小数据存储
+ 对象或大数据请使用数据库或文件存储
+ `apply()` 异步提交，`commit()` 同步提交

## <font style="color:#5C8D07;">【推荐】</font>SharedPreferences 提交数据尽量使用 apply() 而非 commit()
### 说明
+ `Editor#apply()`
    - 先写入内存，然后异步写入磁盘
    - 性能优于 `commit()`，适合频繁写入操作
    - 不会阻塞主线程
+ `Editor#commit()`
    - 直接写入磁盘
    - 返回 `boolean` 表示写入是否成功
    - 阻塞调用线程
    - 仅当需要立即知道提交结果，并据此进行后续操作时才使用

### 正例
```java
// 异步提交，推荐用法
public void updateSettingsAsync() {
    SharedPreferences mySharedPreferences = getSharedPreferences("settings", Activity.MODE_PRIVATE);
    SharedPreferences.Editor editor = mySharedPreferences.edit();
    editor.putString("id", "foo");
    editor.apply();
}

// 同步提交，适用于需要立即获取提交结果的场景
public void updateSettings() {
    SharedPreferences mySharedPreferences = getSharedPreferences("settings", Activity.MODE_PRIVATE);
    SharedPreferences.Editor editor = mySharedPreferences.edit();
    editor.putString("id", "foo");
    if (!editor.commit()) {
        Log.e(LOG_TAG, "Failed to commit setting changes");
    }
}
```

### 反例
```java
// 不推荐频繁使用 commit()
editor.putLong("key_name", "long value");
editor.commit();
```

要点：

+ 默认使用 `apply()` 提升性能，避免阻塞主线程
+ 仅在需要立即获取提交结果时使用 `commit()`

## <font style="color:#FF0000;">【强制】</font><font style="color:#000000;">数据库 Cursor 必须确保使用完后关闭</font>
### 说明
+ `Cursor` 是对数据库查询结果集的管理类。
+ 当查询结果集较大或长时间重复操作时，不关闭 `Cursor` 会导致内存泄漏。
+ 无论查询过程中是否发生异常，都必须在操作完成后确保 `Cursor` 被关闭。
+ 多线程场景下，需要保证 `Cursor` 的操作同步。

### 正例
```java
public void handlePhotos(SQLiteDatabase db, String userId) {
    Cursor cursor = null;
    try {
        cursor = db.query("TUserPhoto", new String[] { "userId", "content" }, "userId=?", 
                          new String[] { userId }, null, null, null);
        while (cursor.moveToNext()) {
            // TODO: 处理查询结果
        }
    } catch (Exception e) {
        // TODO: 异常处理
    } finally {
        if (cursor != null) {
            cursor.close();
        }
    }
}
```

### 反例
```java
public void handlePhotos(SQLiteDatabase db, String userId) {
    Cursor cursor = db.query("TUserPhoto", new String[] { "userId", "content" }, "userId=?", 
                             new String[] { userId }, null, null, null);
    while (cursor.moveToNext()) {
        // TODO: 处理查询结果
    }
    // ❌ Cursor 没有关闭，会导致内存泄漏
}
```

要点：

+ 始终在 `finally` 块中关闭 `Cursor`。
+ 异常情况下也必须关闭。
+ 多线程操作时注意同步。

## <font style="color:#FF0000;">【强制】</font><font style="color:#000000;">多线程操作写入数据库时，需要使用事务</font>
### 说明
+ 通过 `SQLiteOpenHelper` 获取 `SQLiteDatabase` 实例时，Helper 会自动缓存已打开的数据库连接。
+ 单个应用应使用 `SQLiteOpenHelper` 单例模式，确保数据库连接唯一。
+ SQLite 自身是数据库级锁，单个数据库操作保证线程安全，但不能同时进行写入。
+ 使用事务 (`transaction`) 可以保证一组操作的原子性和线程安全，防止多线程写入造成数据库锁异常。
+ 若同时打开多个数据库连接并多线程写入，会导致数据库异常（如“database is locked”）。

### 正例
```java
public void insertUserPhoto(SQLiteDatabase db, String userId, String content) {
    ContentValues cv = new ContentValues();
    cv.put("userId", userId);
    cv.put("content", content);
    
    db.beginTransaction();
    try {
        db.insert("TUserPhoto", null, cv);
        // TODO: 可执行其他相关操作
        db.setTransactionSuccessful();
    } catch (Exception e) {
        // TODO: 异常处理
    } finally {
        db.endTransaction();
    }
}

```

### 反例
```java
public void insertUserPhoto(SQLiteDatabase db, String userId, String content) {
    ContentValues cv = new ContentValues();
    cv.put("userId", userId);
    cv.put("content", content);
    
    // ❌ 未使用事务，多线程写入可能导致数据库锁异常
    db.insert("TUserPhoto", null, cv);
}

```

要点：

+ 多线程写入数据库必须使用事务。
+ 数据库单例模式确保唯一连接。
+ 使用 <font style="color:#5C8D07;">beginTransaction() → 操作 → setTransactionSuccessful() → endTransaction() </font>的顺序。
